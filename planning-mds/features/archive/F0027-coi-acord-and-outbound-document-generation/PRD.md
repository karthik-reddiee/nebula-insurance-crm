# F0027: COI, ACORD & Outbound Document Generation

**Feature ID:** F0027
**Feature Name:** COI, ACORD & Outbound Document Generation
**Priority:** Medium
**Phase:** CRM Release MVP+
**Planning Status:** Phase B proposed; pending Phase B approval

## Feature Statement

**As a** service or distribution user
**I want** Nebula to preview and issue reusable outbound insurance documents from CRM data
**So that** certificates, ACORD forms, and proposal outputs can be prepared inside the audited CRM workflow instead of manually outside the system

## Business Objective

- **Goal:** Add visible insurance-domain output capability to Nebula.
- **Metric:** Users can generate the v1 document set without leaving Nebula; generated artifacts carry template/data provenance and audit history.
- **Baseline:** Nebula can store documents and template files, but not yet generate reusable outbound artifacts.
- **Target:** Users can preview, explicitly issue, retrieve, and regenerate COI, ACORD, and reusable proposal outputs from structured CRM context.

## Problem Statement

- **Current State:** Users prepare certificates, ACORD forms, and proposal outputs outside the CRM, then manually upload or email the result. This weakens traceability and increases risk of stale customer-facing documents.
- **Desired State:** Nebula assembles the v1 outbound document set from approved templates and structured CRM data, stores issued artifacts through the shared document subsystem, and records who generated or issued each artifact.
- **Impact:** Stronger commercial P&C parity, less manual document prep, and better auditability for externally meaningful outputs.

## Personas

- **Service user:** Prepares customer-facing certificates and forms after policy or account context is available.
- **Distribution user:** Generates outbound documents for broker, account, submission, or policy follow-up.
- **Admin:** Governs reusable templates and template versions.
- **Underwriter:** Reviews proposal packet context and may preview proposal output where submission-bound terms already exist.

## Scope & Boundaries

**In Scope for v1:**
- COI, ACORD, and reusable proposal templates.
- Admin-managed template upload/version metadata for those artifact families.
- Preview before issue.
- Explicit Issue action that finalizes an audited generated artifact.
- Structured merge data sourced from authoritative CRM records, including policy, account, broker, submission, and F0019 quote/proposal packet context.
- Generated artifact provenance: source record, template version, generated by, issued by, generated/issued timestamp, artifact family, and draft/final status.
- Storage linkage through the F0020 document system.
- Audit/timeline event on preview failures, successful generation, and final issue where the source record supports timeline events.

**Out of Scope:**
- Submission quote/proposal workflow state, approval readiness, packet status, and bind-decision evidence owned by F0019.
- Full submission intake parsing.
- OCR and extraction.
- E-signature orchestration.
- Real outbound email/send orchestration.
- Rating, pricing, premium calculation, eligibility scoring, or inferred underwriting decisions.
- External broker portal generation flows.

## Success Criteria

- Admin can publish a reusable template for each v1 family: COI, ACORD, and proposal.
- Authorized service/distribution users can preview a generated document before issue.
- Authorized service/distribution users can explicitly issue a final generated artifact.
- Issued artifacts are stored through the shared document subsystem and linked to the source CRM record.
- Regeneration keeps historical issued artifacts immutable and creates traceable new artifacts.
- Missing merge data blocks preview/issue with actionable validation messages.
- All successful issue actions and material failures are auditable.

## Risks & Assumptions

- **Risk:** Output generation is attempted before the underlying data model is ready.
  - **Mitigation:** v1 is sequenced after F0018, F0020, and F0019, all of which are archived as done.
- **Risk:** Bad merge output can become a customer-facing quality issue.
  - **Mitigation:** Require preview before explicit Issue, validate required merge fields, and retain template/data provenance.
- **Risk:** Reusable templates could drift without governance.
  - **Mitigation:** Admin owns template version changes; service/distribution users issue artifacts but do not edit templates in v1.
- **Assumption:** F0020 document storage and template sidecar metadata remain the shared document foundation.
- **Assumption:** F0019 quote/proposal packet terms are recorded, never computed, and reusable proposal rendering consumes them as source data only.

## Dependencies

- F0018 Policy Lifecycle & Policy 360.
- F0020 Document Management & ACORD Intake.
- F0019 Submission Quoting, Proposal & Approval Workflow for submission-bound quote/proposal packet context when reusable proposal rendering is applied to submissions.

F0027 extends the submission-bound artifacts created by F0019 when reusable proposal rendering is needed; it does not become a prerequisite for F0019's quote-to-bind workflow.

## Workflows

### Template Governance

1. Admin opens Template Library.
2. Admin uploads or replaces a template for COI, ACORD, or proposal.
3. Admin validates required metadata and publishes the version.
4. Published template becomes selectable in Generate Document Panel.
5. Prior versions remain visible for provenance and issued artifact history.

### Preview Then Issue

1. Service/distribution user opens a source account, policy, or submission.
2. User opens Generate Document Panel.
3. User selects artifact family and published template version.
4. User clicks Preview.
5. System validates merge data and returns preview or actionable missing-field errors.
6. User clicks Issue.
7. System stores a final generated artifact through F0020 document storage, appends audit/timeline evidence, and shows artifact detail/download.

### Proposal Rendering From F0019 Packet

1. Underwriter or distribution user opens a submission with F0019 packet context.
2. User selects reusable proposal template.
3. System uses recorded packet facts as source data without computing quote values.
4. Preview/Issue follows the same document-generation flow.

## Architecture & Solution Design Notes For Phase B

Phase B must decide the final architecture, API contracts, schema updates, and KG bindings. Phase A requires these constraints:

- Generated artifacts must not bypass F0020 document storage.
- Template editing is admin-governed.
- Issuance is explicit, audited, and separate from preview.
- F0019 remains owner of quote/proposal workflow state; F0027 owns reusable rendering and generated artifacts.
- Proposal rendering consumes recorded packet facts only; no rating, pricing, or eligibility computation is introduced.
